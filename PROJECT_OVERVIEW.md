# 项目说明

## 1. 项目解决的核心痛点

这个项目解决的是“把模糊的网站需求，稳定地变成可运行的 Vue 网站”这件事。

具体痛点有四类：

1. 需求到实现之间断层很大  
   传统流程里，需求、设计、文案、前端代码往往散在不同人手里，容易反复沟通、来回返工。

2. 单次大模型输出不够可控  
   直接让模型一次性生成整站，常见问题是结构混乱、内容空、样式不稳、文件不完整。

3. 站点生成缺少分工  
   一个模型同时负责规划、设计、文案、代码和审查，很容易互相打架，最终只能得到“看起来像网站”的东西。

4. 落地时缺少安全边界  
   生成结果需要严格限制在当前工作区内，避免越界写文件，并且要能本地校验是否可运行。

所以，这个项目的目标不是“再做一个聊天机器人”，而是做一个能把 brief 转成 Vue 站点的工程化流水线。

## 2. 核心逻辑流

整体流程是一个分阶段的长链推理加多 Agent 协作流程。

### 输入

- 网站 brief
- 输出名称
- 技能集合
- 运行模式：API 模式或离线模式

### 推理与协作链路

1. `PlannerAgent`  
   先把 brief 拆成目标、受众、站点结构、验收标准和风险点。

2. `DesignerAgent`  
   产出视觉方向、色彩、排版、响应式规则和交互约束。

3. `ContentAgent`  
   把需求转成更适合首屏展示的文案、SEO 信息和结构化内容。

4. `VueCoderAgent`  
   根据上一步的规划、设计和内容，生成 Vite + Vue 3 项目文件。

5. `ReviewerAgent`  
   从产品完整度、可用性和代码质量角度检查结果，必要时要求修改。

6. `ValidatorAgent`  
   做本地静态校验，检查必需文件、路径安全和基础配置是否完整。

7. `WriterAgent`  
   将通过校验的结果写入当前工作区下的 `generated-sites/`。

### 长链推理是怎么体现的

这里的“长链推理”不是把所有思考塞进一条超长提示词里，而是拆成多个明确阶段：

- 前一个 Agent 的输出，变成后一个 Agent 的输入
- 每一步都产出结构化结果，便于审查和复用
- 发现问题时可以回到 `VueCoderAgent` 做一次修订

这让推理过程更像一条可追踪的生产线，而不是一次性黑盒生成。

### 多 Agent 协作是怎么体现的

项目不是单 Agent 包办，而是按职责拆分：

- Planner 负责“想清楚”
- Designer 负责“长什么样”
- Content 负责“说什么”
- Coder 负责“怎么实现”
- Reviewer 负责“哪里不对”
- Validator 负责“能不能跑”
- Writer 负责“写到哪里”

这种协作方式的好处是：

- 每个角色更专注
- 输出更稳定
- 问题更容易定位
- 后续可以替换单个 Agent，而不用重写整条链路

## 一句话总结

这是一个把网站生成拆成“规划 -> 设计 -> 内容 -> 代码 -> 审查 -> 校验 -> 写入”的多 Agent 工程化流水线，核心价值是把不稳定的自然语言需求，变成可控、可追踪、可落地的 Vue 网站。
